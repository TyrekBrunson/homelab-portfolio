🛡 Pi-hole — Network-Wide DNS Ad Blocker DNS Proxy Network-Wide

Pi-hole is a self-hosted, network-level DNS sinkhole that intercepts and blocks advertisements, trackers, and malicious domains before they ever reach a device on the network. Unlike browser-based ad blockers that only protect a single device, Pi-hole operates at the DNS layer — every device on the local network routes its queries through it, providing blanket protection across phones, laptops, smart TVs, and IoT devices with zero per-device configuration required.

Hosted on an HP Pavilion TS-15 at 192.168.1.161 and serving as the primary DNS resolver for the home network. Currently blocking 41.8% of all DNS queries (27,550 of 65,865 total) across 21 active clients, backed by 1,569,793 domains on block lists. Demonstrates network infrastructure management, DNS architecture, DHCP/DNS integration, and a proactive security posture — directly relevant to network security and sysadmin roles.

Part 1 AD DS Setup & Organizational Structure

Deployed a full Active Directory environment from scratch on a Windows Server 2022 VM hosted on the Proxmox VE hypervisor (Alienware Alpha R1). Installed and configured Active Directory Domain Services (AD DS), promoted the server to a Domain Controller, and established a local domain. Built out a realistic organizational structure with multiple Organizational Units (OUs) modeled after enterprise environments — including geographic containers (USA, Europe, Asia) with nested OUs for Users, Computers, and Servers. Created Security Groups (both built-in and custom) with appropriate group scopes (Global / Universal / Domain Local) and group types (Security and Distribution) to reflect real-world IAM practices. Created and managed user accounts and assigned group memberships across multiple departments (IT, Finance, HR).

Part 2 Group Policy Management (GPO)

Extended the lab by implementing Group Policy Objects (GPOs) using the Group Policy Management Console (GPMC). Applied the correct configuration type (Computer vs. User) and setting type (Policies vs. Preferences) for each scenario — reinforcing the real-world decision-making process used by enterprise IT administrators daily.

Password Policy Computer Config → Policies. Enforced minimum 12-character length, complexity requirements, and 90-day password expiration. Non-modifiable by users.
Drive Mapping User Config → Preferences. Mapped a network share to a consistent drive letter at login. Users retain ability to add additional drives.
Desktop Wallpaper User Config → Policies. Enforced a standardized corporate desktop wallpaper via UNC path. Users cannot override.
Restrict Control Panel User Config → Policies. Fully prohibited user access to Control Panel across all machines in scope.
Disable USB Storage Computer Config → Policies. Blocked all removable storage classes at the system level — enforced and non-overridable by users.
Part 3 GPO Implementation, Domain Join & Testing

Extended the lab by spinning up a Windows 10 Enterprise VM (via VMware Workstation Pro) as a client machine and joining it to the Active Directory domain. Configured the Windows Server with a static IP address and set the loopback address (127.0.0.1) as the primary DNS server — following real-world best practices for domain controllers. Pointed the client VM's DNS to the domain controller's static IP, verified connectivity via ping and nslookup, and successfully joined the client to the domain. Logged in using domain user accounts created in Part 1, confirming AD authentication worked end-to-end.

Implemented all five GPOs from Part 2 by dragging them from the Group Policy Objects container into their respective OUs — user-scoped GPOs (Drive Mapping, Desktop Wallpaper, Restrict Control Panel) under the Users OU, and computer-scoped GPOs (Password Policy, Disable USB Storage) under the Computers OU. Moved the newly joined client machine from the default Computers container into the correct geographic OU and added a machine description per best practice. Forced immediate GPO application using gpupdate /force from the command line rather than waiting for the default 90-minute refresh cycle, then verified each policy was enforced on the client — confirmed by the Control Panel restriction error message appearing on the domain-joined machine.

⚠ Troubleshooting — Settings App Crash

During Part 3, the Windows Settings app would open briefly and immediately close, blocking the network configuration step required to join the client VM to the domain. After investigating DCOM errors (Event ID 10010), the AppX package, and the VM display adapter — the actual root cause was the Restrict Control Panel GPO created in Part 2. On Windows Server, this policy blocks both Control Panel and the Settings app, including for the Administrator account. While troubleshooting, the static IP and DNS were configured via PowerShell as a workaround.

Symptom Settings flashes open and immediately closes; network config step unreachable
Red herrings DCOM Event ID 10010, Basic Display Adapter, missing AppX package — all investigated and ruled out
Root cause NoControlPanel: 1 in HKCU\...\Policies\Explorer — Restrict Control Panel GPO applied to all users including Administrator
Immediate fix Remove-ItemProperty -Path "HKCU:\...\Policies\Explorer" -Name "NoControlPanel"
Proper fix GPMC → Security Filtering: removed Authenticated Users, added Domain Users only — Domain Admins now exempt, users still restricted. Ran gpupdate /force to confirm
Workaround Configured static IP and DNS via New-NetIPAddress and Set-DnsClientServerAddress while Settings was inaccessible
⚠ Troubleshooting — Client Could Not Contact Domain Controller

After configuring the server, the client VM failed to join the domain with the error "An Active Directory Domain Controller for domain ProxMox.local could not be contacted." The client could ping the DC (192.168.1.240) successfully, meaning the network path was fine — the issue was purely DNS. The client's DNS was pointing to the AT&T router (192.168.1.254) instead of the domain controller, so ProxMox.local couldn't be resolved. An IPv6 router address was also interfering before it was disabled via the registry.

Symptom "AD DC for domain ProxMox.local could not be contacted" when attempting to join domain
Red herring IPv6 DNS from router (2600:1700:...) initially intercepting lookups — disabled via registry (DisabledComponents = 255), but root cause was still IPv4 DNS misconfiguration
Root cause Client DNS Servers pointed to 192.168.1.254 (AT&T router via DHCP) — router has no knowledge of ProxMox.local. DC was never consulted.
Fix Manually set client DNS to DC: netsh interface ipv4 set dnsservers "Ethernet" static 192.168.1.240 primary + Google as secondary. Verified with nslookup ProxMox.local resolving through DC.
Domain join Used sysdm.cpl to open Computer Name/Domain Changes directly — bypassed Settings entirely. Entered ProxMox.local and admin credentials. Domain join succeeded.

Password Policy Edited Default Domain Policy — Computer Config → Policies → Windows Settings → Security Settings → Account Policies → Password Policy. Part 7 Windows File Sharing — NTFS, Share Permissions & Inheritance

Implemented real-world file sharing scenarios on the Windows Server domain, covering both share permissions and NTFS permissions — the two independent permission layers that govern network resource access in enterprise environments. Worked through four hands-on activities modeled after actual workplace access control requests, applying the principle of least privilege throughout. Also covered explicit vs. inherited permissions and how to break inheritance on subfolders for granular access control.

⚠ Key Rule When NTFS and Share permissions are combined, the most restrictive applies. A user with Share Full Control but NTFS Read-Only gets Read-Only over the network.
Share Permissions Network-level only (3 options: Read, Change, Full Control). Applied at the folder level via Advanced Sharing → Permissions. Does not cascade to subfolders or apply to local access.
NTFS Permissions File-system level (granular: Full Control, Modify, Read & Execute, List Folder Contents, Read, Write, Special). Applies to both local and network access, cascades to all subfolders and files.
Activity 1 — Marketing Marketing intern needs read-only access to the Marketing folder. Share permissions already had Everyone → Read. Added Marketing Interns group to NTFS with Read only — removed default permissions to enforce least privilege.
Activity 2 — HR HR Group folder must be invisible to all non-HR staff. Removed Everyone from Share permissions, added HR Staff with Full Control. Added HR Staff to NTFS with Full Control — without both layers, subfolder access would still be restricted.
Activity 3 — Vendor Third-party vendor can upload to Vendor Files folder but cannot read existing files. Share: Full Control for Vendor group. NTFS: Write only — removed Read, Read & Execute, and List Folder Contents to prevent the vendor from viewing other files.
Activity 4 — IT / Licenses All IT staff access the Software folder (Full Control via Share + NTFS). Only Senior IT accesses the Licenses subfolder. Disabled inheritance on Licenses subfolder, explicitly granted Senior IT Full Control — breaking the parent's inherited permissions to enforce isolation.
Inherited vs. Explicit Demonstrated that child folders automatically inherit parent NTFS permissions. Used Advanced Security Settings → Disable Inheritance to break the chain on the Licenses subfolder, then applied explicit permissions for Senior IT only.
Part 8 Advanced NTFS Permissions — Inheritance & Effective Permissions

Extended the file permissions work from Part 7 into advanced NTFS concepts: effective permissions, inheritance behavior, and explicit deny. Effective permissions are the actual combined result of explicit permissions, inherited permissions, group memberships, and deny rules — with explicit deny always overriding any inherited or allowed access. Built two hands-on labs from scratch on a Windows Server domain with a Windows 11 client, replicating enterprise access-control scenarios involving nested folder structures and per-user restrictions.

⚠ Key Rule Explicit Deny always wins. If a user belongs to a group with Allow access but also has an explicit Deny (directly or via another group), the Deny takes precedence — no exceptions.
Effective Permissions The actual resulting access a user has, combining explicit permissions, inherited permissions from parent folders, group membership permissions, and deny rules (which override everything else).
Disable Inheritance — Convert Convert inherited permissions to explicit: preserves all currently inherited permissions as standalone explicit entries on the object, but the object no longer receives future changes from the parent. Used when keeping existing access but cutting the inheritance link.
Disable Inheritance — Remove Remove all inherited permissions: wipes every inherited entry, leaving a clean slate that requires manually re-adding all access. Used when starting a subfolder's permissions from scratch.
Lab 1 — Project Subfolder Isolation Created Common\ with subfolders Project\ and Events\, plus a nested Project\Contracts\. Set Common to Full Control for Everyone (Share + NTFS) — confirmed inheritance cascades automatically to all subfolders and new nested folders. Created a Project security group (Global scope). On Project\: disabled inheritance using Convert, removed Everyone, and granted the Project group Full Control — isolating that one subfolder while Events\ retained Everyone's inherited access.
Lab 2 — Explicit Deny (John Doe) Created Project\ with Read/Write for Everyone, plus subfolders Confidential\ and Materials\ — both inherited Everyone's Read/Write. Created user John Doe (member of Employees group). On Confidential\, added John explicitly and set Deny: Read — verified Everyone (including other employees) retained full access while John alone was blocked, and Materials\ remained completely unaffected.
Part 9 Access-Based Enumeration (ABE)

Implemented Access-Based Enumeration (ABE) — a Windows Server feature that hides shared files and folders from users who lack permission to access them, so users only ever see what they're authorized to open. ABE is purely a visibility filter layered on top of existing NTFS and Share permissions; it does not grant or change access, it simply prevents permission-less folders from appearing in the directory listing at all. Built a realistic two-department file server scenario for HR and IT, configured NTFS and Share permissions per department, then enabled ABE and verified that each department's users see only their own folder.

💡 Why It Matters ABE enhances security (users can't probe for folders they can't open), reduces confusion and help desk tickets, and enforces privacy for sensitive departments like HR and Finance — even users who know the exact path can't see or browse to a folder ABE has hidden from them.
ABE Requirements NTFS volumes only — ABE does not function on FAT32/exFAT. Relies entirely on correctly configured NTFS and Share permissions; misconfigured permissions are not corrected by ABE, only hidden behavior is affected.
AD Setup Created two security groups (Global scope): HR Department and IT Department. Created test users — Conan Doyle (added to HR Department) and Lacos (added to IT Department) — via both the "Member Of" tab and group "Members" tab methods.
Shared Folder Structure Created Department Shares\ on the server with subfolders HR\ and IT\ — a single shared root containing department-segregated content.
NTFS Permissions On both HR\ and IT\: disabled inheritance (Convert), removed default Users/Authenticated Users entries, and granted Full Control to the respective department group only — HR Department on HR\, IT Department on IT\.
Share Permissions On Department Shares\ (the parent): removed Everyone, added both HR Department and IT Department groups with Read access — granting entry into the share so users can reach their respective subfolders.
Enabling ABE Server Manager → File and Storage Services → Shares → right-click Department Shares → Properties → Settings → enabled Access-Based Enumeration.
Verification Logged in as Conan (HR) on a Windows 11 client joined to the domain via \\ServerName\Department Shares — only the HR folder was visible, IT was completely hidden. Logged in as Lacos (IT) — only the IT folder was visible, HR was hidden. Confirmed new subfolders created under HR correctly inherited HR-only permissions.

Stage 1 Runtime Environment — Java 25 (Eclipse Temurin)

Installed the Eclipse Temurin JDK 25 distribution (Windows x64 MSI) — the minimum runtime required to host a Hytale server, with both x64 and ARM64 builds supported. Verified the installation and associated .jar file handling so the server binary would launch correctly via Java.

Stage 2 Server Provisioning & OAuth2 Authentication

Used the official Hytale Downloader CLI to fetch the dedicated server binaries and the ~3.3GB Assets.zip game asset package — avoiding the need to manually copy files out of the game client installation. Configured /auth persistence encrypted so credentials are stored encrypted on disk (auth.enc) and survive server restarts without re-authentication.

Built a Start.bat launch script that enables an AOT (Ahead-of-Time) compilation cache for faster startup, points to the server JAR and asset package, and schedules automated backups to a local backup/ directory every 30 minutes. Customized config.json to set the server name, message-of-the-day, max player count (100), view radius (32), and default game mode (Adventure) — keeping server identity and behavior fully version-controllable as plain JSON.

Launch Command java -XX:AOTCache=HytaleServer.aot -jar HytaleServer.jar --assets Assets.zip --backup --backup-dir backup --backup-frequency 30
